## 0. 文档控制

### 0.1 文档目的

本文档将 V5.0 投资策略转化为可开发、可测试、可分阶段交付的产品需求，并结合当前代码能力制定未来迭代计划。

4. 后续按什么顺序开发、验收和发布。

### 0.2 权威文档关系

| 文档 | 权限 | 作用 |
|---|---|---|
| 投资策略说明书 V5.0 | 投资规则最高权威 | 定义目标配置、金额公式、阈值、风控和权限边界 |

发生冲突时：策略金额和阈值以策略说明书为准；页面流程和交互以本 PRD 为准；实现不得通过前端或 AI 改写策略结果。

### 0.3 版本范围

V5.0 不是简单增加 MACD 或现金展示，而是把当前产品升级为：

```text
策略版本 + 冻结输入快照
→ 数据质量门禁
→ 现金账本与权益配置基准
→ 确定性规则计算
→ 资金守恒校验
→ 人工确认执行
→ 成交回填
→ 审计、回放与复盘
```

## 1. 产品背景

### 1.1 用户问题

个人 ETF 长期投资面临的主要问题不是缺少行情，而是信息和决策无法形成稳定闭环：

- 持仓、现金和待投资资金身份容易混淆；
- 行情、估值、净值、溢价和宏观数据质量不稳定；
- 定投、估值增强、QDII 挂起、再平衡规则散落，难以复算；
- 技术指标容易被误用为短线买卖信号；
- AI 解释可能与确定性金额不一致；
- 用户执行后没有回填，系统无法复盘计划与实际差异；
- 策略迭代后缺少版本、回放和审计，无法确认变化影响。

### 1.2 产品定位

产品服务于单个长期投资者，提供“数据可信、规则确定、资金守恒、用户确认、全程可追溯”的 ETF 资金调拨辅助。

产品不是自动交易机器人，也不是预测涨跌的荐股工具。核心产物是：

> **一张可以复算、可以确认、可以回填、可以追溯的每周资金调拨执行单。**

### 1.3 产品目标

1. 将 V5.0 策略完整固化为唯一规则引擎；
2. 确保每笔现金只有一个身份，所有资金流转可还原；
3. 让数据异常明确影响全局、单标的或单模块，不误判也不静默；
4. 让建议从“展示金额”升级为“生成、确认、执行、回填、复盘”的任务；
5. 支持策略版本化、历史回放和变更前后对比；
6. 保留人工最终决策权，不接入未经确认的自动下单。

### 1.4 非目标

- 不预测短期涨跌和收益率；
- 不根据新闻或主观观点自动改仓；
- 不允许 AI 修改金额、标的、阈值和资金去向；
- V5.0 首期不直接连接券商自动下单；
- 不建设多用户、社交、跟单和公开组合功能；
- 不用短期收益率作为规则是否正确的唯一评价标准；
- 不在核心产品中建设复杂宏观研究终端。

## 2. 用户、场景与成功标准

### 2.1 目标用户

当前只服务产品所有者本人，具备以下特征：

- 长期持有多只境内和跨境 ETF；
- 按周投入资金，同时存在挂起现金和再平衡资金；
- 需要规则辅助，但保留最终确认权；
- 愿意校正 OCR、数据异常和实际成交结果；
- 关注可解释性、数据可信度和执行纪律。

### 2.2 核心场景

| 场景 | 用户目标 | 产品结果 |
|---|---|---|
| 每周定投 | 更新持仓和本周注资，生成建议 | 资金调拨执行单 |
| QDII 高溢价 | 不追高且不丢失战略归属 | 专属挂起池及释放计划 |
| 极端高估/超配 | 仅回收超配部分 | 再平衡建议及备用金去向 |
| 数据异常 | 知道哪里错、影响什么 | 分层阻断、降级和修复入口 |
| 周内执行 | 知道立即、分批还是等待 | 技术执行模式，不改变战略金额 |
| 成交后 | 记录真实成交和未成交 | 账本更新及计划偏差 |
| 策略变更 | 了解调整阈值的历史影响 | 版本比较和历史回放 |

### 2.3 北极星指标

**可审计执行闭环完成率**：当周完成“持仓确认 → 数据门禁 → 建议生成 → 用户确认 → 成交回填 → 资金守恒”的周数占计划执行周数比例。

### 2.4 产品指标

| 指标 | V5.0 目标 | 说明 |
|---|---:|---|
| 规则确定性 | 100% | 同一策略版本和输入快照输出完全一致 |
| 总量资金守恒率 | 100% | 每次发布前强制校验 |
| 逐账户资金守恒率 | 100% | 每个现金子账户独立守恒 |
| 硬否决穿透率 | 0 | 被否决标的不得产生场内买入或释放 |
| 核心执行数据有效率 | >= 99% | 按周统计价格、持仓、交易状态等关键输入 |
| 建议可追溯率 | 100% | 金额可回溯到输入、参数和规则命中 |
| AI 金额越权率 | 0 | 不一致文案拒绝发布 |
| QDII 资金串用率 | 0 | 两个挂起池不得互换 |
| 人工确认覆盖率 | 100% | V5.0 不允许未确认自动下单 |

收益率、回撤等投资结果作为长期观察指标，不作为短周期产品验收门。

## 3. 核心原则与权责边界

### 3.1 权责矩阵

| 模块 | 负责 | 不负责 |
|---|---|---|
| 策略配置 | 保存版本化参数和生效范围 | 临时预测市场 |
| 数据服务 | 获取、清洗、缓存、标记质量 | 猜测缺失数据 |
| 规则引擎 | 计算金额、状态和资金去向 | 生成主观投资观点 |
| 技术指标 | 调整增强仓节奏和执行方式 | 创造预算或独立卖出 |
| 宏观模块 | 提示环境和复盘归因 | 直接调整金额 |
| AI | 解释结构化结果 | 重算、覆盖或创造规则 |
| 前端 | 展示、确认、校正和回填 | 二次计算不同金额 |
| 用户 | 确认输入、覆盖和执行 | 被系统替代承担最终决策 |

### 3.2 不可违反的产品约束

1. 规则引擎是金额唯一来源；
2. 数据门禁先于金额计算；
3. 现金身份唯一，转换必须成对记账；
4. 总量与逐账户守恒未通过，不发布执行单；
5. 硬否决不能被基础仓、技术、宏观、AI 或普通人工文案绕过；
6. QDII 挂起资金归属具体 ETF；
7. 技术指标只作用已批准增强仓；
8. 再平衡只卖超配部分；
9. 所有自动建议都要用户确认后才进入“待执行”；
10. 所有参数变更必须有版本、原因、生效时间和确认记录。

## 5. 核心业务闭环

### 5.1 每周主流程

```text
1. 更新持仓
2. 校准 OCR 或手工录入结果
3. 确认本周外部权益注资
4. 查看数据门禁和修复建议
5. 冻结计算输入快照
6. 运行 V5 规则引擎
7. 校验资金守恒和发布门禁
8. 查看执行单、风控和解释
9. 用户确认、调整执行日期或放弃执行
10. 回填成交、未成交和手续费
11. 自动生成现金账本流水及新持仓快照
12. 周度复盘计划与实际差异
```

### 5.2 建议单状态机

```text
draft
→ calculating
→ blocked / ready_for_review
→ confirmed / rejected / expired
→ partially_executed / executed / cancelled
→ reconciled
```

| 状态 | 含义 | 可执行动作 |
|---|---|---|
| `draft` | 输入尚未冻结 | 修改持仓和注资 |
| `calculating` | 正在计算 | 查看进度，不可重复提交 |
| `blocked` | 发布门禁失败 | 修复数据或输入后重算 |
| `ready_for_review` | 可供人工检查 | 确认或拒绝 |
| `rejected` | 用户明确拒绝本次计划 | 保留审计，不产生账本和成交 |
| `confirmed` | 用户确认计划 | 记录实际成交 |
| `partially_executed` | 部分成交 | 补充成交或结束任务 |
| `executed` | 已完成执行 | 进入对账 |
| `expired` | 超过有效期且未确认/未执行 | 资金保持原身份，重新计算新建议 |
| `cancelled` | 用户终止剩余任务 | 已成交部分进入对账，未成交部分保持原身份 |
| `reconciled` | 持仓和账本已更新 | 进入复盘 |

状态机必须按允许迁移表执行，禁止从 `draft` 直接进入 `executed`、从 `rejected/expired` 恢复执行，或从 `reconciled` 回退。需要重新处理时创建新建议单并引用原单。

### 5.3 计算发布门禁

以下任一失败时，状态进入 `blocked`：

- 策略目标比例不等于 100%；
- 账户级必需数据缺失；
- 总量资金守恒失败；
- 任一现金子账户守恒失败；
- 硬否决标的仍有场内金额；
- 金额超过缺口、预算或单标的上限；
- QDII 挂起资金跨标的使用；
- AI 文案改变金额、标的或资金去向；
- 快照或策略版本在计算期间发生变化。

## 6. 产品信息架构

### 6.1 主导航

1. 本周任务状态栏；
2. 持仓与资金输入；
3. 数据门禁摘要；
4. 本周资金调拨执行单；
5. 风控与再平衡；
6. 现金子账户流水；
7. 执行确认与成交回填；
8. 历史记录入口。

### 6.3 趋势页结构

- ETF 选择器；
- 估值与样本窗口；
- 日 K / 周 K 切换；
- MACD、20 周和 40 周趋势；
- 当前技术原子状态和最终分类；
- QDII 溢价或红利股息率专项指标；
- 数据时间、来源、质量和代理标记；
- 该标的本周规则命中与执行模式。

### 6.4 设置页结构

设置页按职责分区：

1. 策略版本与目标配置；
2. 周预算和当周默认值；
3. 现金账户与释放计划；
4. 规则参数，只允许受控编辑；
5. 数据源、刷新任务和质量；
6. 通知；
7. 后台运维。

数据源日志、数据库浏览和清缓存留在后台运维，不占用核心用户任务首屏。

## 7. Epic 与详细需求

### 7.1 E1：策略版本与冻结快照（P0）

#### 用户故事

作为用户，我希望每次建议都明确使用哪个策略版本和哪组输入，确保以后能够复算。

#### 功能需求

- 新建 `strategy_version`，至少包含版本号、状态、参数 JSON、策略文档引用、生效时间、创建原因和确认人；
- 状态为 `draft/active/retired`，同一时间只能有一个 active 版本；
- 六只核心 ETF 和默认比例作为 V5.0 默认参数，不硬编码为永久产品约束；
- 区分 `strategy_weekly_budget` 与 `weekly_contribution_committed`；
- 每次计算保存策略参数快照、持仓快照、现金快照和数据快照哈希；
- 计算期间任一快照变化时，本次结果作废并提示重算；
- 人工覆盖记录规则、原因、生效/失效时间和覆盖前后值。
- 黑名单、停牌、严重数据冲突和账户级数据错误默认不可人工覆盖；其他覆盖必须由用户显式确认。AI 无权创建覆盖。

#### 验收标准

- 目标比例不等于 100% 时不能激活版本；
- 同一版本、同一快照重复计算结果一致；
- 历史计算不受后来参数修改影响；
- 人工覆盖可查询、可到期，不由 AI 创建。

### 7.2 E2：数据质量门禁（P0）

#### 用户故事

作为用户，我希望知道数据是否能用于交易计算，以及异常影响哪个范围。

#### 功能需求

- 所有策略输入统一输出 `source/as_of/freshness/quality_status/reason`；
- 质量状态统一为 `valid/degraded/stale/conflict/missing`；
- 支持全局阻断、单标的阻断、禁用估值增强、技术回退、仅取消宏观提示五种作用域；
- 快速刷新和深度刷新改为后台任务，返回任务 ID、步骤、成功数、失败数、耗时和重试入口；
- 用户可从异常项直接跳到数据血缘或刷新任务；
- 数据门禁结果写入计算快照，不允许计算后动态漂移。

#### 验收标准

- 账户级异常不发布任何金额；
- 单标的价格冲突不误伤其他标的；
- 技术数据缺失时系数回退 1.00；
- 宏观缺失不影响金额；
- 过期数据不得以“最后成功值”静默参与强规则。

### 7.3 E3：现金账本与权益配置基准（P0）

#### 用户故事

作为用户，我希望清楚知道每笔钱当前属于哪里、为什么流入、何时可以使用。

#### 功能需求

- 建立现金账户：`daily_cash`、`weekly_unallocated_cash`、`rebalance_equity_reserve`、两个 QDII 挂起池、`manual_cash`、`weekly_contribution_committed`；
- 每个账户保存期初、流入、流出和期末；
- 账户转换生成成对借贷流水，流水不可直接覆盖，只能冲正；
- 计算 EAB 时纳入所有仍承诺投入权益的资金；
- 计算合成持仓时将对应 QDII 挂起资金计入战略归属；
- 总量和逐账户守恒均通过后才能发布；
- 总览展示账户余额、当周变化、来源事件和未来释放计划。

#### 验收标准

- 每笔现金只属于一个账户；
- 任意流水可追溯到建议、成交或人工调整；
- 手工制造未扣减挂起池的买入时，守恒校验必须失败；
- 平衡组合新增资金后能够产生新目标缺口；
- 卖出资金不会造成权益配置基准被动缩小。

### 7.4 E4：V5 规则引擎与资金分配（P0）

#### 用户故事

作为用户，我希望获得严格按 V5 目标缺口、基础仓、增强仓和风控计算的唯一结果。

#### 功能需求

- 引擎版本从 `target-gap-rebalance-v4.2` 升级为 V5；
- 依次计算目标市值、合成持仓、新增买入缺口；
- 外部新增预算执行 40% 基础仓、60% 增强仓分桶；
- 普通未分配现金、QDII 挂起和再平衡备用金按独立释放规则计算；
- 支持 PE/PB 保守分位、A500 代理、红利反向分位和 QDII 底层指数规则；
- 严格区分硬否决、普通软风控和极端估值软停止；
- 单标的外部资金不超过周注资 70% 和缺口上限；
- 舍入、最小交易单位和未分配尾差必须有明确去向；
- 输出每步中间值，前端只展示，不重算。

#### 验收标准

- 六只 ETF 比例、边界值和分桶测试全部通过；
- 95% 与 `>95%` 产生不同且唯一结果；
- 普通高估仍可保留基础仓，极端软停止基础和增强均为 0；
- 技术系数大于 1 时总增强仓仍不超预算；
- 所有金额能够由审计字段复算。

### 7.5 E5：QDII 挂起与释放（P0）

#### 用户故事

作为用户，我希望高溢价时不追高，同时保证资金不会被挪用或永久锁死。

#### 功能需求

- 当前溢价 `>3%` 且 3 日均值 `>2.5%` 时，场内买入和释放为 0；
- 保留该标的名义基础额和增强额，进入专属挂起池；
- 建立 `idle/releasing/paused/completed` 持久化状态机；
- 首次释放初始化 8 个有效释放周；
- 零注资周仍按 `strategy_weekly_budget` 计算释放上限；
- 暂停周不减少剩余周数；
- 余额归零后清空计划，下一轮重新初始化；
- 合成暴露余量使用非负上限；
- 连续挂起 4 周提示替代方案，12 周触发人工复核。

单周释放金额必须按策略公式计算，且不受新增买入缺口封顶：

```text
planned_release = pending_balance / max(1, release_weeks_remaining)

actual_release = min(
  planned_release,
  pending_balance,
  strategy_weekly_budget × target_ratio × 2.0,
  max(0, EAB × (target_ratio + 5pp) - current_etf_value)
)
```

#### 验收标准

- 两个 QDII 挂起池不可串用；
- 挂起资金不重复扩大缺口；
- 挂起余额可以最终归零，不出现除零和负释放；
- 阻断、恢复、再次阻断、新增挂起等状态可完整回放；
- 内部释放不重复消耗外部周预算。
- 即使新增买入缺口为 0，只要挂起余额和合成暴露余量允许，仍能按计划完成现金向 ETF 的形态转换。

### 7.6 E6：技术执行与周内任务（P1）

#### 用户故事

作为用户，我希望知道本周建议应该立即、分批、等待还是仅执行基础仓，而不是仅看到一个 MACD 金叉/死叉。

#### 功能需求

- 获取复权日 K 和周 K；
- 计算 MACD `12/26/9`、20 周和 40 周均线；
- 保存周线原子状态、日线原子状态和最终互斥分类；
- 输出 `strong/conflict/very_weak/improving/weak/neutral/unavailable`；
- 映射唯一系数和 `immediate/staged/wait_pullback/base_only`；
- 趋势页展示指标如何形成最终状态；
- 总览把执行模式转成周内任务，例如一次执行、分两次、等待确认；
- 技术数据缺失时回退中性，不阻断基础规则；
- QDII 使用底层指数信号，不用境内溢价价格替代。

#### 验收标准

- 分类互斥且完备；
- 金叉后快速死叉归入冲突，不错误放大；
- 技术层不能使零缺口变正，也不能独立卖出；
- `base_only` 只清零增强仓；
- 科创 50 强势状态仍默认分批，不追单日急涨。

### 7.7 E7：再平衡与备用金（P1）

#### 用户故事

作为用户，我希望只在长期极端高估且明显超配时卖出，并确保卖出资金继续服务权益配置。

#### 功能需求

- 按 A 股宽基、红利、海外 QDII 分别执行 V5 阈值；
- 多档同时命中只执行最高档；
- 卖出金额不超过超配金额；
- 周线强势最多允许观察一周；
- 卖出款进入 `rebalance_equity_reserve`；
- 卖出当周不重新投入，自下一周按策略周预算和缺口释放；
- 总览展示“为什么卖、卖多少、钱去哪、何时再投入”；
- 再平衡建议必须独立确认和成交回填。

自动化边界测试必须覆盖：A 股估值分位恰好 90%、95% 及其上下相邻值；红利股息率分位恰好 10%、15% 及其上下相邻值；QDII 底层估值和场内溢价边界；超配恰好 5、8、10 个百分点及其上下相邻值；各档 20%、30%、50% 卖出比例；强趋势观察一周后仍命中时强制执行。

#### 验收标准

- 无超配时估值再高也不卖；
- 多档命中不叠加比例；
- 技术指标不能提高卖出额；
- 零注资周备用金仍可释放；
- 卖出款不会进入日常现金。

### 7.8 E8：执行确认与成交回填（P1）

#### 用户故事

作为用户，我希望把建议转成可管理的执行任务，并记录真实成交结果。

#### 功能需求

- 建议单支持整体确认，也支持逐项拒绝并填写原因；
- 记录计划金额、计划份额、计划日期和执行模式；
- 回填实际成交价格、份额、金额、手续费和时间；
- 支持部分成交、未成交、撤销和过期；
- 未成交资金按原身份回退，不自动改变账户；
- 成交后生成新持仓快照和账本流水；
- 计划与实际差异在周度复盘中展示；
- V5.0 不提供自动券商下单按钮。

#### 验收标准

- 未确认建议不能进入成交状态；
- 部分成交后剩余资金身份正确；
- 手续费进入资金守恒；
- 成交回填后持仓、现金和执行单一致；
- 重复提交成交回填具备幂等保护。

### 7.9 E9：审计、回放与复盘（V5.0 P1 / V5.1 P2）

#### 用户故事

作为用户，我希望能够解释任何一次建议，并在修改策略前看到历史影响。

#### 功能需求

- 保存 V5 最低审计字段；
- 支持按计算 ID 查看输入、步骤、规则命中、资金流水和 AI 校验；
- 支持使用历史快照重放指定策略版本；
- 新旧策略对同一快照并行计算，但不写入真实账本；
- V5.0 提供周度执行复盘及月/季/年复盘所需的完整底层数据；
- V5.1 再提供月度资金、季度数据与挂起、年度策略复盘看板；
- 统计纪律执行率、数据有效率、挂起周期、现金拖累和偏离修复；
- AI 只解释结构化结果，一致性失败时显示模板解释和失败原因。

#### 验收标准

- 任一执行单可完整重放；
- 回放不污染生产现金和持仓；
- 策略变更可以对比金额、标的、挂起和卖出差异；
- AI 不可修改任何结构化结果；
- 审计记录不可由普通 UI 直接删除。
- V5.0 周度复盘可追踪，月/季/年所需数据可查询；V5.1 看板另行验收。

## 8. 数据与接口需求

### 8.1 新增核心领域对象

| 对象 | 关键字段 |
|---|---|
| `strategy_version` | version, status, parameters, doc_ref, effective_at, confirmed_by |
| `calculation_snapshot` | strategy_version, holdings_hash, cash_hash, market_hash, frozen_at |
| `cash_account` | account_type, assigned_asset, opening_balance, current_balance, status |
| `cash_ledger_entry` | debit_account, credit_account, amount, source_event, reference_id, occurred_at |
| `release_plan` | plan_type, account_id, state, weeks_total, weeks_remaining, balance |
| `execution_order` | calculation_id, asset, side, planned_amount, mode, status |
| `execution_fill` | order_id, price, shares, amount, fee, executed_at, idempotency_key |
| `manual_override` | rule, before, after, reason, effective_at, expires_at, confirmed_by |
| `replay_run` | source_calculation, strategy_version, result, diff, created_at |

### 8.2 API 分组

| API | 用途 |
|---|---|
| `/api/strategy-versions` | 创建、激活、查询策略版本 |
| `/api/calculations` | 冻结快照并生成 V5 计算 |
| `/api/calculations/{id}` | 获取执行单和完整审计 |
| `/api/cash/accounts` | 查询账户余额和释放计划 |
| `/api/cash/ledger` | 查询流水，受控人工调整 |
| `/api/release-plans` | 查询 QDII 和备用金状态机 |
| `/api/execution-orders` | 确认、拒绝和管理执行任务 |
| `/api/execution-fills` | 回填成交并触发对账 |
| `/api/replays` | 运行历史回放和版本对比 |
| `/api/data-jobs` | 运行和查询快速/深度刷新任务 |

### 8.3 接口原则

- 所有写接口支持幂等键；
- 所有金额使用整数分或 Decimal，不用浮点累计账本；
- 计算接口返回结构化中间结果，不只返回最终金额；
- 前端不得通过本地公式重算金额；
- 长任务返回任务 ID，不让 HTTP 请求等待数分钟；
- 所有响应包含版本、时间和追踪 ID；
- 历史对象优先追加和冲正，不物理覆盖。

高风险写接口必须覆盖重复提交测试：策略激活、人工覆盖、账本调整、建议确认/拒绝、释放计划迁移和成交回填在相同幂等键下只能生效一次；不同请求体复用同一幂等键必须拒绝。

## 9. 异常与降级

| 异常 | 产品行为 | 用户操作 |
|---|---|---|
| OCR 低置信度 | 标红字段，不生成最终快照 | 人工校正 |
| 账户总额不一致 | 全局阻断 | 检查持仓和现金 |
| 单标的价格冲突 | 单标的硬否决 | 刷新或人工确认数据 |
| 估值缺失 | 禁用该标的增强仓 | 可继续基础仓 |
| 技术数据缺失 | 技术回退 1.00 | 无需阻断 |
| 宏观缺失 | 隐藏相关提示 | 不影响金额 |
| 数据刷新超时 | 保留已成功子任务 | 重试失败项 |
| AI 服务不可用 | 使用确定性模板 | 仍可执行规则结果 |
| 守恒失败 | 不发布执行单 | 查看流水差异 |
| 成交部分失败 | 保持部分执行状态 | 补充或结束任务 |
| 策略变更中 | 禁止激活第二版本 | 完成或取消变更 |

## 10. 非功能需求

### 10.1 正确性

- 金额计算和账本使用确定性数值类型；
- 所有阈值边界必须有自动化测试；
- 状态机迁移必须拒绝非法跳转；
- 计算、回放和前端展示使用同一结构化结果。

### 10.2 性能

- 使用缓存时，打开总览页首屏目标 `< 2 秒`；
- 使用已冻结数据生成建议目标 `< 5 秒`，不含 AI；
- 快速刷新按后台任务执行，单项结果实时可见；
- 深度刷新允许分钟级，但不得阻塞页面和建议历史查看。

### 10.3 可用性

- 数据服务故障时仍能查看最后一次执行单和审计；
- AI 故障不影响规则结果；
- 刷新任务可重试失败项，不重复拉取成功项；
- 重要操作有明确的进行中、成功、失败和恢复状态。

### 10.4 安全

- 数据源 Token 加密存储，不写入日志；
- 人工覆盖、策略激活、账本调整和数据清理需要二次确认；
- 审计记录默认只追加；
- 本地部署默认不暴露公网；
- 未来券商接入必须单独立项和安全评审。

### 10.5 可观测性

- 每次刷新、计算、AI 解释、确认和成交都有 trace ID；
- 后台展示任务耗时、失败步骤、数据源和重试次数；
- 提供规则版本、数据版本和引擎版本；
- 关键失败可通过通知模块提醒，但不得包含密钥和完整账户敏感信息。

- 不允许负释放、串池和重复计资；
- 本阶段通过后，V5 只进入“影子计算灰度”：与 V4.2 并行展示差异，不允许确认、成交或写入真实账本。

### 11.6 Sprint 3：技术执行工作台

**范围**

- E6 技术分类器和趋势页展示；
- E8 建议确认状态机前半部分；
- 执行模式转周内任务；
- 快速/深度刷新后台任务化；
- 完善移动端和桌面端核心流程。

**验收门**

- 技术状态互斥且完备；
- 技术层无金额越权；
- 用户可确认或拒绝建议；
- 长刷新不阻塞页面；
- 浏览器桌面和移动视口无关键布局/交互问题。

### 11.7 Sprint 4：再平衡与成交闭环

**范围**

- E7 再平衡和备用金完整释放；
- E8 成交回填、部分成交、手续费和对账；
- 更新持仓快照和现金账本；
- 周度计划与实际差异。

**验收门**

- 卖出不超过超配金额；
- 卖出资金 100% 进入备用金；
- 成交回填幂等；
- 部分成交和未成交资金身份正确；
- 计划、实际、持仓和账本四方一致。

### 11.8 Sprint 5：审计回放与正式发布

**范围**

- E9 的 V5.0 范围：完整审计、历史重放、策略版本对比和周度复盘；
- AI 结构化一致性拒发；
- 全量迁移、回归和浏览器验收；
- V4.2 引擎只读保留，V5 成为默认；
- 更新用户说明和运维手册。

**验收门**

- V5 全局验收场景全部通过；
- 任意执行单可重放；
- 回放不污染真实账本；
- AI 越权率为 0；
- 数据备份、迁移和回滚演练通过。

### 11.9 V5.1 及后续

## 12. 全局验收场景

### 12.1 策略与数据

- [ ] 目标比例不是 100% 时拒绝激活策略；
- [ ] 相同策略和快照输出完全一致；
- [ ] 账户数据缺失时全局阻断；
- [ ] 单 ETF 数据冲突只阻断该标的；
- [ ] 估值、技术、宏观缺失分别按权限降级；
- [ ] 样本 499、500、999、1000 的结果唯一。

### 12.2 买入与风控

- [ ] 平衡组合新增 7,000 元后产生正缺口；
- [ ] 普通高估保留基础仓；
- [ ] 95% 与 `>95%` 边界正确；
- [ ] 基础/增强 40%/60% 分桶守恒；
- [ ] 单标的不超过外部周注资的 70%；
- [ ] 全部软风控、全部软停止和全部硬否决结果不同且唯一。

### 12.3 现金与 QDII

- [ ] 每个子账户满足期初加流入减流出等于期末；
- [ ] 总资金来源等于买入、挂起、留存和其他去向；
- [ ] QDII 高溢价金额只进入对应挂起池；
- [ ] 零注资周仍可释放内部资金；
- [ ] 释放不会产生负金额或超出合成暴露；
- [ ] 新一轮挂起重新初始化计划。

### 12.4 技术与再平衡

- [ ] 日线确认和走弱双命中归入冲突；
- [ ] 技术数据缺失回退 1.00；
- [ ] 技术系数不能突破预算和缺口；
- [ ] 再平衡只卖超配部分；
- [ ] 多档命中只执行最高档；
- [ ] 强趋势最多延迟一周。
- [ ] A 股、红利和 QDII 的估值、超配、溢价边界及卖出比例逐项通过自动化测试。

### 12.5 执行与审计

- [ ] 未确认建议不能记录成交；
- [ ] 部分成交、手续费和撤销均守恒；
- [ ] 重复回填不会重复记账；
- [ ] AI 改写金额时文案拒发、规则单保留；
- [ ] 任意历史计算可重放；
- [ ] 策略对比不污染真实持仓和现金。
- [ ] 建议单非法状态跳转被拒绝，拒绝、过期和取消后的资金身份正确。
- [ ] 所有高风险写接口重复提交只生效一次，幂等键复用冲突被拒绝。

## 13. 风险与依赖

| 风险 | 影响 | 应对 |
|---|---|---|
| 外部数据源慢或不稳定 | 刷新耗时、数据缺失 | 后台任务、缓存、分层降级、官方校验源 |
| 当前数据库分散 | 账本和行情审计难统一 | 明确领域库边界和引用 ID，不在 Sprint 1 大迁移 |
| 现有规则自动化测试很少 | V5 边界回归风险高 | Sprint 0 先建立规则、账本、状态机和 API 测试基线 |
| 历史规则硬编码 | 新旧结果混用 | 策略版本和引擎版本强制写入 |
| 浮点金额误差 | 守恒误报或漏报 | 使用分/Decimal 和统一舍入策略 |
| 状态机复杂 | 资金锁死或重复释放 | 状态迁移表、属性测试和回放 |
| 技术指标过拟合 | 频繁调整规则 | 固定 V5 参数，变更必须版本化和回放 |
| AI 服务不可用或幻觉 | 解释失败或越权 | 模板回退、结构化校验、AI 无阻断权 |
| 单人开发范围过大 | 迭代拖延 | 每个 Sprint 设置退出条件，P2 不进入 P0 |
| 浏览器自动化环境不稳定 | UI 验收缺证据 | 保留 Playwright 可重复路径和人工验收清单 |

### 13.1 待产品决策

以下参数在进入对应 Sprint 前必须确认。产品所有者负责业务口径，开发负责人负责技术可行性和默认建议：

- 执行数据最大允许时效；
- “流动性严重不足”的量化阈值；
- 最小交易金额、交易单位和尾差规则；
- 建议单默认有效期；
- 部分成交未完成金额的回退时点；
- 手工账本调整的授权和冲正流程；
- 数据库存储是否继续 SQLite，何时需要迁移；
- V5.0 正式切换时历史现金余额的迁移基准日。

冻结时间：执行数据时效、流动性、交易单位、尾差和不可覆盖规则在 Sprint 0 结束前冻结；建议有效期、部分成交回退、账本授权和冲正在 Sprint 2 开始前冻结；数据库边界和现金迁移基准日在 Sprint 1 结束前冻结。未冻结的阻断项不得带入依赖 Sprint。

## 14. 发布与回滚

### 14.1 灰度策略

1. V5 引擎先在固定快照和回放环境运行；
2. 与 V4.2 并行计算两周，只展示差异，不写真实账本；
3. Sprint 3 完成人工确认后，V5 可进入“仅确认、不回填”的演练环境，仍不得用于真实执行；
4. Sprint 4 完成成交与对账闭环后，才允许小范围真实周度执行；
5. 完成两次真实周度执行和对账后，V5 进入建议默认、V4.2 只读并发布；
6. V4.2 历史记录永久保留，不重算覆盖。

### 14.2 回滚条件

出现任一情况立即停止 V5 新建议：

- 资金守恒失败；
- QDII 串池或重复释放；
- 硬否决被穿透；
- 同快照输出不确定；
- 成交回填重复记账；
- 数据迁移导致持仓或现金无法核对。

回滚只切换默认计算版本，不删除 V5 已产生的审计记录和账本流水；错误流水通过冲正处理。

## 15. 最终交付定义

V5.0 完成不是页面出现“技术指标”或“资金池”几个新字段，而是同时满足：

```text
策略有版本
输入有快照
数据有门禁
现金有身份
计算有唯一结果
执行有人工确认
成交有回填
资金能守恒
历史能重放
AI不越权
```

最终用户应当能够在一次周度流程中回答：

1. 本周有哪些钱可以投；
2. 为什么分配给这些 ETF；
3. 哪些资金因什么原因被挂起；
4. 应立即、分批还是等待执行；
5. 是否需要卖出超配部分；
6. 实际成交后钱和持仓去了哪里；
7. 任何一个结果能否从数据、参数和规则完整还原。
